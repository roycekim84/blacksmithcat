# AGENTS.md - 대장장이 고양이 구현 지침

## 1. 프로젝트 개요

프로젝트명: 대장장이 고양이
저장소: https://github.com/roycekim84/blacksmithcat
플랫폼: 모바일 앱
화면 방향: 세로형
장르: 캐주얼 무기 강화 게임
그래픽 스타일: 2D 픽셀아트

## 2. 핵심 컨셉

신비한 능력을 가진 대장장이 고양이가 무기를 강화하고, 강화된 무기를 팔아서 골드와 재료를 모으는 게임이다.

현재 MVP는 에피소드 1이며, +1부터 +20까지의 강화 플레이를 구현한다.
장기적으로는 +99강까지 확장할 수 있도록 데이터 구조를 설계한다.

## 3. 개발 우선순위

이 프로젝트는 작은 모바일 게임으로 실제 출시 가능한 완성도를 목표로 한다.
처음부터 거대한 시스템을 만들지 말고, `강화해서 팔기` 루프를 먼저 완성한다.

가장 중요한 순서:
1. 세로형 모바일 UI
2. 강화 버튼과 성공/실패 처리
3. 골드/재료 소모
4. 무기 판매
5. 무기 분해
6. 상점
7. 저장/불러오기
8. 연출과 사운드
9. 광고/BM 연동 준비

## 4. 추천 기술 방향

Royce는 주로 Flutter 기반으로 모바일 앱/게임을 만드는 흐름을 선호한다.
따라서 별도 지시가 없다면 Flutter 프로젝트로 구현한다.

권장 구조:
- Flutter stable
- 세로 고정
- 상태 관리는 초기에는 단순 Provider/Riverpod/ValueNotifier 중 하나로 충분
- 데이터는 JSON 또는 Dart map/table 기반으로 분리
- 저장은 shared_preferences 또는 로컬 JSON 기반으로 시작

게임 엔진이 꼭 필요하지 않다. 화면 전환과 간단한 애니메이션 중심이므로 Flutter 기본 위젯/애니메이션으로 MVP 구현 가능하다.

## 6. MVP 기능 명세

### 6.1 메인 강화 화면
필수 표시:
- 현재 골드
- 주요 재료 수량
- 현재 무기 이름
- 현재 강화 단계
- 성공 확률
- 강화 비용
- 실패 패널티 설명
- 판매 예상가
- 강화 버튼
- 판매 버튼
- 분해 버튼
- 상점 이동 버튼
- 기록/설정 이동 버튼

### 6.2 강화 기능
입력:
- 현재 무기 강화 단계
- 보유 골드
- 보유 재료
- 선택한 보호권

처리:
- 강화 비용과 재료가 충분한지 확인
- 비용 차감
- 성공률에 따라 성공/실패 결정
- 성공 시 강화 단계 +1
- 실패 시 구간별 패널티 적용
- 결과 연출 표시
- 저장

구간:
- +1 ~ +5: 실패 시 비용/재료만 소모
- +6 ~ +10: 낮은 확률로 -1
- +11 ~ +15: -1 가능
- +16 ~ +20: -1 및 낮은 확률 파손 가능

### 6.3 +10 이후 다른 강화 방식
+10 이후에는 `마력 주입 강화`로 표시한다.
UI 문구와 필요 재료가 바뀐다.

- +1 ~ +10: 일반 강화
- +11 ~ +20: 마력 주입 강화

### 6.4 판매 기능
현재 무기를 판매하여 골드를 획득한다.
판매 후 새 기본 무기 또는 시작권 기반 무기를 준비한다.

+10 이상 무기는 판매 전 확인 팝업을 띄운다.

### 6.5 분해 기능
현재 무기를 분해하여 재료를 얻는다.
판매보다 골드는 적거나 없지만 재료를 얻는다.

+10 이상 무기는 분해 전 확인 팝업을 띄운다.

### 6.6 상점
구매 가능 상품:
- 철 조각
- 마력석
- 검 조각
- 하락 방지권
- 깨짐 방지권
- +3 시작권
- +5 시작권

초기에는 실제 결제 없이 게임 내 골드 구매만 구현한다.
광고 보상 버튼은 자리만 만들어도 된다.

### 6.7 기록 화면
표시:
- 최고 강화 단계
- 총 강화 시도 수
- 총 성공 수
- 총 실패 수
- 총 판매 골드
- +20 달성 여부

### 6.8 설정 화면
- 사운드 ON/OFF
- 진동 ON/OFF
- 데이터 초기화

## 7. 데이터 설계 원칙

강화 확률, 비용, 판매가, 필요 재료는 코드 로직에 직접 박지 말고 테이블로 분리한다.

예시 필드:

```dart
class EnhancementRule {
  final int targetLevel;
  final double successRate;
  final int goldCost;
  final Map<String, int> materialCost;
  final FailureType failureType;
  final int sellPrice;
}
```

실제 타입명은 자유롭게 정하되, 데이터 기반으로 쉽게 조정 가능해야 한다.

## 10. 밸런싱 기준

첫 플레이에서 +10까지는 비교적 쉽게 도달해야 한다.
+20은 하루 안에 가능하되, 너무 즉시 끝나면 안 된다.

권장 감각:
- +5: 1~3분 내 도달
- +10: 5~10분 내 도달
- +15: 15~30분 내 도달
- +20: 반복 도전 후 달성

정확한 수치는 플레이 테스트로 조정한다.

## 11. UX 주의사항

- 강화 결과 연출은 길게 만들지 않는다.
- 실패 후에도 바로 다시 강화할 수 있어야 한다.
- 재료 부족 시 어떤 재료가 부족한지 즉시 보여준다.
- 판매/분해 실수 방지를 위해 고강화 무기는 확인 팝업을 둔다.
- 확률과 실패 패널티는 숨기지 않는다.

## 12. 아트 임시 처리

초기 구현 시 실제 픽셀아트 리소스가 없으면 다음 방식으로 임시 처리한다.

- 고양이: 간단한 사각형/이모지/임시 PNG
- 무기: 단계별 색상이 다른 간단한 도형
- 배경: 대장간 느낌의 단색/그라디언트 박스
- 이펙트: Flutter 애니메이션으로 임시 구현

아트가 없다는 이유로 기능 구현을 멈추지 않는다.

## 13. 금지 사항

- 처음부터 복잡한 전투 시스템을 만들지 않는다.
- 여러 무기 타입을 MVP에 넣지 않는다.
- 결제 시스템을 먼저 붙이지 않는다.
- 강화 테이블을 하드코딩 로직에 박지 않는다.
- 가로형 UI를 만들지 않는다.
- 문서에 없는 대형 기능을 임의로 추가하지 않는다.
